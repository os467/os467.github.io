分布式基础理论

什么是分布式系统？

分布式系统是若干独立计算机的集合，这些计算机对于用户来说就像个相关系统

分布式系统（distributed system）是建立在网络之上的软件系统

随着互联网的发展，网站应用的规模不断扩大，常规的垂直应用架构已无法应付，分布式服务架构以及流动计算架构势在必行，急需一个治理系统确保架构有条不紊的进行

参考博客

单一应用架构

当网站流量很小时，只需一个应用，将所有功能都部署在一起，以减少部署节点和成本，此时，用于简化增删改查工作量的数据访问框架（ORM）是关键

适用于小型网站，小型管理系统，将所有功能都部署到一个功能里，简单易用

缺点：性能扩展比较难，协同开发问题，不利于升级维护

垂直应用架构

当访问量逐渐增大，单一应用增加机器带来的加速度越来越小，将应用拆成互不相干的几个应用，以提升效率，此时，用于加速前端页面开发的Web框架（MVC）是关键

通过切分业务来实现各个模块独立部署，降低了维护和部署的难度，团队各司其职更易管理，性能扩展也更方便，更有针对性

缺点：公用模块无法重复利用，开发性的浪费

分布式服务架构

当垂直应用越来越多，应用之间交互不可避免，将核心业务抽取出来，作为独立的服务，逐渐形成稳定的服务中心，使前端应用能更快速的响应多变的市场需求，此时，用于提高业务复用及整合的分布式服务框架（RPC）是关键

流动计算架构

当服务越来越多，容量的评估，小服务资源的浪费等问题逐渐显现，此时需增加一个调度中心基于访问压力实时管理集群容量，提高集群利用率，此时，用于提高机器利用率的资源调度和治理中心（SOA）Service Oriented Architecture 是关键

搭建分布式架构会用到的技术栈：

spring cloud 基于springboot的框架，springboot可以看成是分布式的一个节点

  • 具有一栈式的解决方案

dubbo + zookeeper

  • 没有很多组件，集成第三方组件较为轻松，例如nginx

RPC

什么叫RPC？

RPC【Remote Procedure Call】是指远程进程调用，是一种进程间通信方式，它是一种技术的思想，而不是规范，RPC允许程序调用另一个地址空间（通常是共享网络的另一台机器上）的过程或函数，而不用程序员显式编码这个远程调用的细节，即程序员无论是调用本地的还是远程的函数，本质上编写的调用代码基本相同

RPC基本原理

客户端会将需要发送的对象序列化成加密数据，通过sockets通讯网络发送到服务端，服务端接收到数据后反序列化成需要的对象

dubbo核心概念

简介

Apache Dubbo 是一款高性能、轻量级的开源Java RPC 框架，它提供了三大核心模块： 面向接口的远程方法调用，智能容错（服务健康检测、服务权重）和负载均衡，以及服务自动注册和发现

官网：http://dubbo.apache.org/

基本概念

服务提供者(Provider) ：暴露服务的服务提供方，服务提供者在启动时，向注册中心注册自己提供的服务

服务消费者(Consumer) ：调用远程服务的服务消费方，服务消费者在启动时，向注册中心订阅自己所需的服务，服务消费者，从提供者地址列表中，基于软负载均衡算法，选一台提供者进行调用，如果调用失败，再选另一台调用

注册中心(Registry) ：注册中心返回服务提供者地址列表给消费者，如果有变更，注册中心将基于长连接推送变更数据给消费者

监控中心（Monitor）：服务消费者和提供者，在内存中累计调用次数和调用时间，定时每分钟发送一次统计数据到监控中心

调用关系说明

  • 服务容器负责启动，加载，运行服务提供者

  • 服务提供者在启动时，向注册中心注册自己提供的服务

  • 服务消费者在启动时，向注册中心订阅自己所需的服务

  • 注册中心返回服务提供者地址列表给消费者，如果有变更，注册中心将基于长连接推送变更数据给消费者

  • 服务消费者，从提供者地址列表中，基于软负载均衡算法，选一台提供者进行调用，如果调用失败，再选另一台调用

  • 服务消费者和提供者，在内存中累计调用次数和调用时间，定时每分钟发送一次统计数据到监控中心

dubbo环境搭建

【windows】安装zookeeper

1、下载zookeeper

网址：https://archive.apache.org/dist/zookeeper/zookeeper-3.4.13/

2、解压zookeeper

解压运行zkServer.cmd，初次运行会报错，没有zoo.cfg配置文件

3、修改zoo.cfg配置文件

将conf下的zoo_sample.cfg复制一份改名为zoo.cfg即可

注意几个重要位置：

dataDir=./ 临时数据存储的目录（可写相对路径）

clientPort=2181 zookeeper的端口号

修改完成后再次启动zookeeper

4、使用zkCli.cmd 测试

【Linux】安装zookeeper

linux安装java环境：

yum list java*

yum install java-1.8.0-openjdk* -y

关闭防火墙

systemctl stop firewalld

开机关闭防火墙

systemctl disable firewalld

显示文件列表

ll

解压zookeeper包

tar -zxvf zookeeper-3.4.13.tar.gz

进入bin目录

cd zookeeper-3.4.13

cd bin

启动zkServer

/zkServer.sh start

进入conf文件夹

cd conf

将dubbo-admin.war 移动到 /apache-tomcat-8.0.53/webapps/目录下部署服务，重启zookeeper

在tomcat的bin目录下启动startup.sh

访问注册中心http://192.168.119.88:8080/dubbo-admin/

springboot整合dubbo

创建两个工程，一个作为请求资源的客户端，一个作为传输资源的服务端

创建dubbo_client工程、dubbo_server工程

导入maven坐标

<!--dubbo场景启动器-->
<dependency>
    <groupId>io.dubbo.springboot</groupId>
    <artifactId>spring-boot-starter-dubbo</artifactId>
    <version>1.0.0</version>
</dependency>

dubbo提供者配置

spring.dubbo.application.name=dubbo-server
spring.dubbo.registry.address=zookeeper://(ip地址):2181
spring.dubbo.protocol.name=dubbo
spring.dubbo.protocol.port=20880
spring.dubbo.scan=com.*.impl

注：暴露服务的扫描包需要定位到具体的服务包impl下

dubbo消费者配置

spring.dubbo.application.name=dubbo-client
spring.dubbo.registry.address=zookeeper://(ip地址):2181
spring.dubbo.scan=com.*.controller

仓库

创建一个maven工程，用来存放Service资源

通过clean+install命令将当前工程打包存放到maven仓库中，再通过当前工程的maven坐标在客户端Controller获取到Service资源

注：保证在idea中用的是同一个仓库

客户端表现层

package com.os467.demo.controller;

import com.os467.service.TestService;
import org.springframework.stereotype.Controller;
import org.springframework.web.bind.annotation.RequestMapping;

@Controller
@RequestMapping("test")
public class TestController {

    private TestService testService;

}

导入maven仓库中之前保存的Service资源

<dependency>
    <groupId>org.example</groupId>
    <artifactId>dubbo_demo</artifactId>
    <version>1.0-SNAPSHOT</version>
</dependency>

服务消费者

导入dubbo提供的注解 @Reference

@Reference 用于订阅服务

 @Reference
 private TestService testService;

服务提供者

服务端也要导入坐标并且实现Service接口

服务端需要去暴露服务

使用dubbo提供的**@Service**注解

@Service 用于暴露服务

package com.os467.demo.service.impl;

import com.alibaba.dubbo.config.annotation.Service;
import com.os467.service.TestService;

@Service
public class TestServiceImpl implements TestService {

    @Override
    public void findData() {

        System.out.println("消费者调用提供者调用成功");

    }

}

修改下服务端端口号，防止报错

server.port=8081

客户端继续用默认的8080端口

访问视图资源，虽然会报错status=500（没有配置视图），但是我们可以在服务端控制台上发现消费者调用提供者调用成功，说明本次调用成功

在分布式项目中，由于架构分离扫描不到持久层接口，我们需要使用@Mapper注解
